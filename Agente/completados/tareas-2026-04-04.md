- **Que:** Los tags OpenAPI contenían caracteres no-ASCII (`Campañas`, `Gestión Restaurante API`) que al pasar por Orval en Windows generaban directorios/archivos con encoding corrupta (`campa├▒as/`, `gestiNRestauranteAPI.schemas.ts`) y duplicados.   - `frontend/src/api/generated/campanas/` — Renombrado (antes campañas/campa├▒as)

  - 5 archivos manuales (hooks/componentes) — Imports actualizados de `gestiNRestauranteAPI` a `gestionRestauranteAPI`

## 044A-3 — Fix "No se pudo conectar con el servidor" + eliminar datos de prueba
- **Que:** El frontend mostraba "No se pudo conectar con el servidor" al intentar eliminar datos porque leía `err.response.data.mensaje` pero el backend envía `err.response.data.message` (campo `message` en `ErrorResponse`). Se corrigió el campo. Además, el endpoint `DELETE /api/admin/reset` solo borraba 12 tablas — se expandió a 20 tablas cubriendo todas las relaciones FK. Se agregó `DEMO_MODE=true` al stack de producción (requerido para que el endpoint funcione).
- **Archivos modificados:**
  - `src/handlers/admin.rs` — `eliminar_datos` expandido de 12 a 20 DELETEs (agregados: `combinacion_mesa_items`, `campana_destinatarios`, `recordatorios_enviados`, `reservas_etiquetas`, `clientes_etiquetas`, `notificaciones`, `reglas_recordatorio`, `campanas`)

  - `coolify-manager-rs/templates/rust-stack.yaml` — Agregado `DEMO_MODE: 'true'` al environment
- **Gotchas:**
  - El `ErrorResponse` de Axum usa campo `message` (inglés), pero `AdminResult` de éxito usa `mensaje` (español). Diferencia sutil que causaba el bug.

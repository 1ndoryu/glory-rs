## 044A-2 — Fix encoding corrupta en archivos generated (Orval)
- **Que:** Los tags OpenAPI contenían caracteres no-ASCII (`Campañas`, `Gestión Restaurante API`) que al pasar por Orval en Windows generaban directorios/archivos con encoding corrupta (`campa├▒as/`, `gestiNRestauranteAPI.schemas.ts`) y duplicados. Se cambiaron los tags a ASCII puro en el backend Rust, se regeneró el OpenAPI JSON y todo el código Orval.
- **Archivos modificados:**
  - `src/handlers/campanas.rs` — 7 ocurrencias de `tag = "Campañas"` → `tag = "Campanas"`
  - `src/handlers/mod.rs` — `title = "Gestión Restaurante API"` → `title = "Gestion Restaurante API"`, `VentaConCliente` agregado a schemas
  - `frontend/src/api/generated.ts` — Barrel re-export actualizado a nuevos nombres
  - `frontend/src/api/generated/gestionRestauranteAPI.schemas.ts` — Renombrado (antes gestiN/gestión)
  - `frontend/src/api/generated/campanas/` — Renombrado (antes campañas/campa├▒as)
  - Todos los archivos en `frontend/src/api/generated/*/` — Regenerados con imports a `gestionRestauranteAPI`
  - 5 archivos manuales (hooks/componentes) — Imports actualizados de `gestiNRestauranteAPI` a `gestionRestauranteAPI`
- **Gotchas:**
  - `dump_openapi` vía PowerShell genera BOM UTF-8 si usas `Out-File -Encoding UTF8`. Usar `[System.IO.File]::WriteAllText` con `UTF8Encoding($false)`.
  - `VentaConCliente` no estaba en el `schemas()` de `ApiDoc` aunque `VentasPaginadas` lo referenciaba — Orval fallaba con "Ref not found".
  - Los archivos corruptos con `├▒` eran realmente encodings dobles de UTF-8 (bytes `c3 b1` interpretados como Windows-1252).
- **Sentinel:** No aplica
  - Path resolution del config: binary está en `target/release/`, hay que subir 3 niveles para llegar al root de coolify-manager-rs.
  - `rust_vars()` ya no recibe DB_PASSWORD/JWT_SECRET — esos se dejan como `${VAR}` en el template para que Coolify los resuelva. Solo DOMAIN, GLORY_BRANCH, REPO_URL son template vars.
- **Sentinel:** No aplica (cambio cross-project, no frontend/backend específico)
- **Testeado:** `.\scripts\deploy.ps1 -SkipBuild -SkipComposeSync` → Deploy exitoso, health check OK (status 200)

## 044A-3 — Fix "No se pudo conectar con el servidor" + eliminar datos de prueba
- **Que:** El frontend mostraba "No se pudo conectar con el servidor" al intentar eliminar datos porque leía `err.response.data.mensaje` pero el backend envía `err.response.data.message` (campo `message` en `ErrorResponse`). Se corrigió el campo. Además, el endpoint `DELETE /api/admin/reset` solo borraba 12 tablas — se expandió a 20 tablas cubriendo todas las relaciones FK. Se agregó `DEMO_MODE=true` al stack de producción (requerido para que el endpoint funcione).
- **Archivos modificados:**
  - `src/handlers/admin.rs` — `eliminar_datos` expandido de 12 a 20 DELETEs (agregados: `combinacion_mesa_items`, `campana_destinatarios`, `recordatorios_enviados`, `reservas_etiquetas`, `clientes_etiquetas`, `notificaciones`, `reglas_recordatorio`, `campanas`)
  - `frontend/src/componentes/Configuracion.tsx` — Catch block: `err.response.data.mensaje` → `err.response.data.message`
  - `coolify-manager-rs/templates/rust-stack.yaml` — Agregado `DEMO_MODE: 'true'` al environment
- **Gotchas:**
  - El `ErrorResponse` de Axum usa campo `message` (inglés), pero `AdminResult` de éxito usa `mensaje` (español). Diferencia sutil que causaba el bug.
  - El deploy vía Coolify API falló (HTTP 500 en compose sync, SSH timeout en build). Resolución: SSH directo al servidor + `docker compose up -d --build`.
  - Coolify restart dejó contenedores en estado "exited" sin reconstruirlos. No confiar en restart para deploys que requieren rebuild.
- **Sentinel:** No aplica
- **Testeado:** Login + `POST /api/admin/reset` → `{"ok":true,"mensaje":"Todos los datos de prueba han sido eliminados."}`. Verificado: ventas=0, clientes=0, reservas=0.

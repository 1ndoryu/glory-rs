## 294A-3 - Fix procesos scraping (proxy 407 + warm-up Cloudflare)
- **Que:** Los procesos de scraping fallaban con `407 ProxyError` (credenciales DataImpulse expiradas). Ademas se descubrio que Cloudflare Managed Challenge bloquea IPs sucias del pool rotativo: cuando el warm-up obtiene 403, WhoSampled sirve una pagina sin contenido (0 entries `li.listEntry`) aunque el HTTP status sea 200. Fix: (1) PROXY_PASSWORD actualizado en `.env` (no commiteado — gitignored), (2) warm-up refactorizado para reintentar hasta 5 veces con nuevas sesiones curl_cffi (= nuevas IPs DataImpulse) hasta obtener HTTP 200 limpio, fijando esa sesion para todos los requests del spider, (3) warning en spider si page 1 devuelve 0 entries, (4) verificar_proxy() y deteccion 407 en cron_runner.
- **Root cause CF:** La IP del proxy rotativo determina si Cloudflare desafia o no. Con IP "limpia" → warmup 200 → hot-samples 200 con 20 entries. Con IP "sucia" → warmup 403 (managed challenge) → hot-samples 200 pero contenido vacio (JS-hydrated shell). Los selectores `li.listEntry` NO habian cambiado — confirmado inspeccionando HTML real en proxy con IP limpia.
- **Archivos:** `clients/kamples-scraper/kamples_scraper/middlewares.py`, `clients/kamples-scraper/kamples_scraper/spiders/hot_samples.py`, `clients/kamples-scraper/scripts/cron_runner.py`, `clients/kamples-scraper/.env` (local only).
- **Commit:** `6b41aab` (warm-up retry) + `0ca1aef` (per-request session refresh) rama `kamples`.
- **Gotchas:** `nodriver` (Chrome real headless) tampoco pasa CF Managed Challenge en headless. El enfoque correcto es reintentar el warm-up hasta obtener IP limpia. DataImpulse NO soporta sticky sessions via `-session-ID` en username (devuelve 407 NO_USER). Cada request curl_cffi abre nuevo CONNECT tunnel = nueva IP del pool rotativo, por eso el primer fix (solo warm-up) NO bastaba: warm-up obtenia IP limpia, pero la siguiente request del spider salia por otra IP y obtenia 403. Segundo fix: `_fetch` ahora reintenta hasta 4 veces ante 403, llamando `_refresh_session()` entre intentos para forzar nueva IP+cookie.
- **Validacion:** Sintaxis OK. Run real del spider tras segundo fix: `'downloader/response_status_count/200': 16`, cero 403, 6 items scraped, 200 entries totales (198 nuevas).
- **Sentinel:** No aplica — cambios son en Python scraper, no en codigo Rust.
- **GLORY-RS:** No aplica.
## 294A-4 - Fix visibilidad logs procesos + scraping bloqueado + publicacion incremental
- **Que:** Reportes desde la UI: (1) extraccion mostraba estado=running pero log_tail vacio en TarjetaProceso, (2) scraping no arrancaba ("Tarea daily ya en ejecucion") por lock huerfano, (3) samples solo se publicaban al final del lote (si el proceso se interrumpia, quedaban en estado extraido sin publicar).
- **Archivos:** `src/services/admin_processes.rs`, `clients/kamples-scraper/scripts/cron_runner.py`, `clients/kamples-scraper/extractor/pipeline.py`.
- **Commit:** `3216f5e` rama `kamples`.
- **Root cause logs vacios:** `read_log_tail` construia el path con `Utc::now().format("%Y-%m-%d")` y devolvia "" si no existia. Cuando el archivo de log fue creado en otra fecha (TZ shift UTC vs local, proceso largo cruzando medianoche, o simple drift) el tail aparecia vacio aunque el proceso siguiera escribiendo.
- **Root cause lock:** `adquirir_lock` solo invalidaba locks por edad >6h. Tras crash sin liberar lock, scraping quedaba bloqueado horas. Ahora lee el PID del lock y verifica vivacidad con `OpenProcess` (Windows) / `os.kill(pid, 0)` (Unix); si el PID esta muerto, elimina el lock inmediatamente.
- **Root cause publicacion:** `notificar_publicacion(N)` se llamaba solo al final del bucle `for item in pendientes`. Ahora se llama con `1` tras cada item exitoso, dentro de try/except para no romper el lote si falla la publicacion.
- **Validacion:** `cargo check` OK; `cargo build --bin glory-backend` OK (Finished 2m 38s, binario 88MB en C:\tmp\glory-target\debug\); `python -m extractor.pipeline --limit 2` end-to-end con 9 exitosos/11 fallidos (fallidos por YouTube bot detection, conocido sin cookies); 87 samples publicados manualmente via /api/admin/scraper/publicar-auto durante diagnostico; 11 items huerfanos recuperados de descargando/analizando -> pendiente.
- **Sentinel:** No requiere regla nueva.
- **GLORY-RS:** No aplica. Cambios son nakomi/kamples-specific (scraper Python + servicio admin de procesos).

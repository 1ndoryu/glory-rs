- **Sentinel:** No aplica.
- **GLORY:** No aplica — integración específica entre el backend de Nakomi y el API de Coolify para crear hostings WordPress de clientes.

## 165A-3 — Compose SSH parseable para provisioning de hosting

- **Que:** El `500` restante en producción no venía del contrato HTTP sino del compose generado para el sidecar SSH/SFTP: Coolify no podía parsear el `dockerfile_inline` y abortaba el alta cerca del `FROM lscr.io/linuxserver/openssh-server...`. Se reescribieron los builders de SSH con literales YAML claros, se eliminó basura de diff que había quedado en el compose estático, y se añadieron pruebas que parsean el YAML generado tanto para hosting WordPress como para hosting normal.
- **Archivos:** `src/services/coolify.rs`, `Cargo.toml`, `Cargo.lock`, `roadmap.md`, `Agente/documentacion/hosting/checkout-publico-hosting-2026-04-10.md`, `Agente/lecciones/lecciones-aprendidas.md`
- **Gotchas:** Las pruebas anteriores solo buscaban strings y no detectaban que el compose completo podía ser inválido para un parser YAML real. El guard nuevo usa `serde_yaml` en tests para romper antes de llegar a Coolify.
- **Validacion real:** Tras desplegar `studio`, un `POST /api/hosting/subscribe` autenticado como `test@test.com` creó la suscripción `87052c71-2dcb-42bf-86ad-4fd7b46638a6` ya con `server_uuid`, `server_ip`, `sftp_user` y `sftp_port`. Además, las suscripciones huérfanas previas `38f4cb0b-f8af-4b1f-88f4-34a2fc9fb038` y `701de52d-b551-4a83-8e36-b29e0166eee7` quedaron rellenadas por el backfill del panel.
- **Sentinel:** No aplica.
- **GLORY:** No aplica — fix de generación de compose para stacks de hosting creados por el backend de Nakomi.

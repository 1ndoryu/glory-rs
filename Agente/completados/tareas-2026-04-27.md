## 274A-1 - Corrige render JSON en AdminPanelIsland

- **Que:** Se corrigio el crash de React en `AdminPanelIsland` cuando la pestaña de moderación intentaba renderizar `moderacion_detalle` dentro de `TarjetaHistorial`. El helper de render ahora convierte cualquier `moderacion_detalle` a string segura antes de pintarlo en `<pre>`, y el tipo frontend se ajusto de `string | null` a `unknown` para reflejar que el backend puede entregar JSON ya parseado.
- **Archivos:** `frontend/src/legacy/components/admin/TabModeracionAdmin.tsx`, `frontend/src/legacy/services/apiAdmin.ts`, `Agente/documentacion/frontend/admin-moderacion-json-2026-04-27.md`
- **Gotchas:** La traza apuntaba al `setState` del hook `useAdminPanel`, pero la causa real estaba en el presenter `TarjetaHistorial`. El navegador del agente no tenia sesion admin, asi que la validacion se hizo con la traza del usuario + `npx tsc --noEmit`.
- **Sentinel:** No aplica.
- **GLORY-RS:** No aplica. Es comportamiento especifico del panel admin de Kamples.

## 274A-2 - Corrige listado de publicaciones en comunidad y perfil

- **Que:** Se corrigio el desajuste entre el contrato Rust de `GET /api/publicaciones` y los hooks legacy que consumen ese listado. El backend ya responde con `{ items, page, per_page }`, pero comunidad y perfil seguian leyendo un shape antiguo basado en `data`, así que convertían la respuesta a lista vacía aunque existieran publicaciones. Ambos hooks ahora normalizan `items`, `data` y arrays directos.
- **Archivos:** `frontend/src/legacy/hooks/useComunidadIsland.tsx`, `frontend/src/legacy/hooks/usePerfilIsland.ts`, `Agente/documentacion/frontend/publicaciones-listado-items-2026-04-27.md`
- **Gotchas:** La landing pública de `/comunidad/` oculta el feed cuando no hay sesión, así que la verificación visual desde el navegador del agente era ambigua. La comprobación decisiva fue `curl http://localhost:3000/api/publicaciones?page=1`, que confirmó que el backend devolvía posts reales bajo `items`.
- **Sentinel:** No aplica.
- **GLORY-RS:** No aplica. Es compatibilidad específica entre este frontend legacy y el contrato del backend Kamples.

## 274A-3 - Normaliza PostDetail Rust para TarjetaPublicacion legacy

- **Que:** Se corrigió el crash de runtime en `TarjetaPublicacion` dentro de comunidad. El backend Rust ya entregaba publicaciones reales, pero en shape `PostDetail` snake_case; la UI legacy esperaba `Publicacion` camelCase. Se agregó un normalizador central de publicaciones en `apiSocial` para adaptar campos como `autor_id`, `samples_adjuntos`, `created_at`, `mi_reaccion` y `yo_ya_repostee` antes de que lleguen a los componentes.
- **Archivos:** `frontend/src/legacy/services/normalizers/postNormalizer.ts`, `frontend/src/legacy/services/apiSocial.ts`, `frontend/src/legacy/hooks/useComunidadIsland.tsx`, `Agente/documentacion/frontend/publicaciones-normalizacion-rust-legacy-2026-04-27.md`
- **Gotchas:** El IDE no lo marcaba porque era una discrepancia de payload real en runtime, no un error estático. La primera corrección de `items` vs `data` permitió por fin renderizar posts y eso expuso el segundo fallo en `samplesAdjuntos.length`.
- **Sentinel:** No aplica.
- **GLORY-RS:** No aplica. Es adaptación puntual entre el contrato Rust de Kamples y la UI legacy del mismo proyecto.

## 274A-4 - Permite a admin borrar publicaciones ajenas

- **Que:** Se corrigió la 403 al intentar borrar publicaciones ajenas desde comunidad con un usuario admin. `DELETE /api/publicaciones/:id` exigía autoría exacta y bloqueaba incluso a admins; ahora permite borrar si el usuario tiene `rol == "admin"`, manteniendo intacta la restricción de no usar este endpoint para reposts.
- **Archivos:** `src/handlers/posts.rs`, `Agente/documentacion/backend/admin-delete-posts-2026-04-27.md`
- **Gotchas:** `soft_delete()` necesita el `autor_id` real del post, no el `user_id` del admin. Por eso el nuevo guard devuelve `owner_id` y el handler lo reutiliza al persistir. La búsqueda textual no encontró esos nombres smoke en los archivos del repo, así que todo apunta a residuos de smoke tests previos en la BD local y no a fixtures versionados.
- **Sentinel:** No aplica.
- **GLORY-RS:** No aplica. Es política de permisos específica del backend Kamples.

## 124A-IMAGE404 — Fix permanente: imágenes 404 por pérdida de bind mount

- **Que:** Coolify normaliza CUALQUIER bind mount path a named volume en su API interna. Cuando escribe el compose a disco (restart desde UI, redeploy, auto-restart), reemplaza el bind mount `/data/uploads/studio:/app/uploads` con un named volume `UUID_uploads-data:/app/uploads`. Docker crea un named volume vacío → imágenes perdidas.
  
  **Solución:** `volume_manager::ensure_uploads_bind_mount()` — módulo compartido que ejecuta `sed` en el compose en disco después de cada operación de Coolify, forzando el bind mount correcto. Se ejecuta en `deploy_service`, `redeploy` y `restart_site`.

  Patrón sed: `'[^']*:/app/uploads'` → `'/data/uploads/{site}:/app/uploads'`
  - Cubre named volumes Coolify (`UUID_uploads-data`)
  - Cubre bind mounts incorrectos
  - Es idempotente cuando ya está correcto

- **Archivos:**
  - `coolify-manager-rs/src/services/volume_manager.rs` (nuevo)
  - `coolify-manager-rs/src/services/mod.rs`
  - `coolify-manager-rs/src/commands/deploy_service.rs`
  - `coolify-manager-rs/src/commands/redeploy.rs`
  - `coolify-manager-rs/src/commands/restart_site.rs`

- **Gotchas:**
  - La API de Coolify NO soporta `file_storages` para bind mounts via API (endpoints no existen)
  - Coolify `docker_compose_raw` almacena `uploads_data:/app/uploads` (normalizado)
  - El compose en disco puede diferir completamente del raw/processed de la API
  - El test del sed desde PowerShell→SSH causa problemas de escape; el ssh client Rust envía el comando directamente a bash sin intermediario
  - 5 imágenes de proyectos están permanentemente perdidas (UUIDs en BD no existen en disco)
  - Si alguien reinicia desde Coolify UI (no coolify-manager-rs), el compose se reescribe con named volumes. El PRÓXIMO deploy desde coolify-manager-rs lo corregirá.

- **Sentinel:** Posible regla para detectar compose files con named volumes para `/app/uploads` en templates. Baja prioridad.

- **GLORY-RS:** No aplica — la lógica es específica de Coolify deploy, no del framework.

- **Root cause chain (documentar para futuras referencias):**
  1. Template `rust-stack.yaml` tiene bind mount correcto
  2. `sync_compose` PATCH a Coolify API → Coolify normaliza a named volume
  3. Coolify escribe compose procesado a disco con named volume
  4. `docker compose up -d` usa named volume → contenedor nuevo = uploads vacío
  5. Fix: `ensure_uploads_bind_mount()` sed-reemplaza after Coolify writes

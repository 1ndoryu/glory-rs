## 064A-5 — Sincronización unidireccional con Haddock POS API
- **Qué:** Implementación completa de sincronización automática de ventas con Haddock POS API. Backend: servicio `haddock.rs` con cliente HTTP, mapeo de canales/métodos de pago, retry exponencial (3 intentos), hooks post-create/update en VentaService y ReservaService (ventas automáticas). Migración: campos `haddock_api_token` + `haddock_sync_enabled` en configuracion_restaurante. Frontend: token API + toggle de sincronización en la sección Haddock de Configuración.
- **Archivos:** `src/services/haddock.rs` (nuevo), `src/services/mod.rs`, `src/services/venta.rs`, `src/services/reserva.rs`, `src/models/configuracion.rs`, `src/repositories/configuracion.rs`, `migrations/20260406000000_haddock_sync.up.sql` + `.down.sql`, `.sqlx/` (3 archivos actualizados), `frontend/src/hooks/useConfiguracion.ts`, `frontend/src/componentes/Configuracion.tsx`, `frontend/openapi-debug.json`, `frontend/src/api/generated/gestionRestauranteAPI.schemas.ts`
- **Gotchas:** El token API usa `#[serde(skip_serializing)]` para no exponerlo en respuestas GET, similar a groq_api_key. Cache SQLx offline requirió actualización manual (3 archivos JSON) porque no hay BD local. El hash SHA256 de la nueva UPDATE query se computó con PowerShell. Haddock requiere `items[]` obligatorio pero nuestra plataforma no desglosa artículos — se envía un único ítem genérico con nombre=descripcion. Haddock requiere ISO 8601 con hora pero solo tenemos fecha — se estima hora por turno (mañana=09:00, mediodía=14:00, noche=21:00).
- **Sentinel:** No aplica.
- **Pendiente de prueba real:** No se puede probar contra API real hasta que el cliente proporcione su token. Tests unitarios cubren mapeo de datos.

# Tareas completadas — 2026-04-25

## 254A-1 — Restaurar 3 procesos legacy en admin (scraping, extraccion, seed)

- **Que:** El tab de Procesos en admin mostraba "No se encontraron procesos configurados" porque el backend Rust no exponía las rutas `/api/admin/procesos*` que el cliente legacy (`apiProcesos.ts`) consume vía `wpJsonStub`. Se creó el servicio `AdminProcessService` con los 3 procesos hardcodeados (scraping, extraccion, seed) — equivalente al comportamiento PHP que siempre devolvía los 3 aunque no hubiera locks. Se agregaron handlers, rutas y modelos OpenAPI para `list_processes`, `process_state`, `start_process`, `stop_process`, `process_cookies` (GET/POST).
- **Archivos:**
  - `src/services/admin_processes.rs` (nuevo, ~489 líneas) — servicio con enum `ProcessKind`, lock files, log tail, cookie info para yt-dlp.
  - `src/services/mod.rs` — registro del módulo.
  - `src/models/admin.rs` — structs `AdminProcessState`, `AdminProcessesResponse`, `AdminProcessStartRequest`, `AdminProcessCookiesRequest`, `AdminProcessCookieInfo`.
  - `src/models/mod.rs` — re-exports.
  - `src/handlers/admin.rs` — 6 handlers nuevos con `#[utoipa::path]` para codegen.
  - `src/handlers/mod.rs` — rutas + registro en `ApiDoc`.
  - `openapi.json` — regenerado.
- **Validacion:** `cargo clippy -- -D warnings` limpio · `cargo build --bin glory-backend` OK · `npx tsc --noEmit` (frontend) limpio · `GET /api/admin/procesos` sin token responde **401** (ruta montada correctamente, requiere `require_admin`). El frontend legacy ya consume estos endpoints sin cambios — su `apiProcesos.ts` hace fetch a `/wp-json/kamples/v1/admin/procesos*` y `wpJsonStub.ts` los reescribe a `/api/admin/procesos*`.
- **Gotchas:**
  - El binario `glory-backend.exe` quedó bloqueado en el target principal mientras un proceso del usuario lo tenía abierto; resuelto cerrando ese proceso y compilando con `SQLX_OFFLINE=true`.
  - Para emitir `openapi.json` no es necesario `cargo run` — basta `cargo build` + ejecutar el binario directamente con `--emit-openapi`.
  - `apiProcesos.ts` NO usa Orval, así que no se necesitó codegen frontend para esta tarea (solo verificar tsc).
- **Sentinel:** No aplica — ningún patrón nuevo a detectar.
- **GLORY-RS:** No aplica — `AdminProcessService` es específico de Kamples (procesos `scraping`/`extraccion`/`seed` con paths/log files propios del proyecto). No es agnóstico.

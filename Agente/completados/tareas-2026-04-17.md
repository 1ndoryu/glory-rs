## 164A-16 — Seguridad hosting: Fases 2-3 + port collision + headers
- **Qué:** Ejecutadas Fases 2-3 del Plan de Seguridad Hosting. (1) Port collision: migración UNIQUE en sftp_port + `find_available_sftp_port()` con retry 10x en rango 10000-49151. (2) Network isolation: 3 redes (frontend_net, backend_net internal, ssh_net) — MariaDB solo accesible por WP, SSH aislado. (3) Container hardening: cap_drop ALL + cap_add mínimos, security_opt no-new-privileges, pids_limit (200 WP, 150 MariaDB, 100 SSH), memory reservations. (4) WP hardening: imagen pineada wordpress:6.7-php8.3-apache, MariaDB 11.4, openssh-server tag específico, DISALLOW_FILE_EDIT. (5) Security headers: añadidos Referrer-Policy y Permissions-Policy al middleware.
- **Archivos:** `coolify.rs`, `hosting.rs` (repo + handler), `hosting_stripe.rs`, `mod.rs` (handlers), migration `20260417000000_hosting_sftp_port_unique.*`, `.sqlx/` cache
- **Gotchas:** El compose template no puede ser una sola función por límite clippy 100 líneas — dividido en `build_compose_wp_db()` + `build_compose_ssh()`. Firma de `provision_hosting()` cambió para recibir `sftp_port` externamente (generado por caller con acceso a pool).
- **Sentinel:** No aplica
- **GLORY-RS:** La lógica de compose hardening + port allocation es específica de Nakomi hosting. No aplica a glory-rs.

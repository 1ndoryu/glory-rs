## 224A-5 — Fase3-sqlx: convertir sqlx::query() a query!() para validación compile-time
- **Que:** se convirtieron todas las llamadas a `sqlx::query()` sin macro que usan SQL estático a `sqlx::query!()` para que SQLx verifique los tipos y la existencia de tablas/columnas en tiempo de compilación. Se actualizó el cache `.sqlx/` con `cargo sqlx prepare`.
- **Archivos:** `src/repositories/moderation.rs` (5 queries), `src/repositories/user.rs` (1), `src/handlers/scraper_admin.rs` (1), `src/algorithm/precompute.rs` (comentarios explicativos, no convertible — DDL REFRESH MATERIALIZED VIEW). 5 nuevos archivos en `.sqlx/`.
- **No convertibles (documentado con comentarios):**
  - `precompute.rs`: `REFRESH MATERIALIZED VIEW` — PostgreSQL no soporta preparación de DDL de mantenimiento via extended query protocol. Debe quedarse como `query()`.
  - `glory-rs/fixtures/mod.rs` line 87: `CREATE TABLE IF NOT EXISTS` — DDL, no preparable.
  - `glory-rs/fixtures/mod.rs` line 218: `&delete_sql` — SQL dinámico (tabla/columna variables), imposible con macro.
  - `glory-rs/fixtures/sync.rs` + `mod.rs` (DELETE _glory_fixtures): la tabla `_glory_fixtures` es creada en runtime por `ensure_tracking_table()` — no existe en DB local al momento de `cargo sqlx prepare`.
  - `user.rs` + `moderation.rs` varios `query_as(&sql)`: SQL dinámico construido con `format!()`.
- **Gotchas:** `SQLX_OFFLINE=true` en `.env` significa que TODOS los `query!()` necesitan entrada en `.sqlx/`. `cargo sqlx prepare` ignora este flag y conecta a la DB real. La DB local `glory_kamples` en `localhost:5432` debe estar corriendo para poder regenerar el cache.

## 224A-4 — fix imágenes home y modal auth no cargan (URLs absolutas WP)
- **Que:** en el SPA (browser/producción), las imágenes del landing y del modal de auth generaban URLs absolutas tipo `https://kamples.com/wp-content/themes/glorytemplate/App/Assets/...` que 404 porque el backend Rust no sirve esa ruta. El resolver debía retornar `/legacy-assets/...` que nginx sí sirve desde el build Vite.
- **Causa raíz:** `resolverRutaAsset.ts` usaba `__KAMPLES_DESKTOP__` como proxy de "es Tauri". Pero `gloryContextShim.ts` pone ese flag en `true` en todos los entornos (incluyendo browser web) para activar auth JWT Bearer. - **Fix:** cambiar `esEntornoTauri()` para usar `window.__TAURI_INTERNALS__` — el global que Tauri v2 inyecta únicamente en su WebView. En browser normal siempre es `undefined`, en Tauri desktop siempre está presente.
- **Archivos:** `frontend/src/legacy/utils/resolverRutaAsset.ts`.
- **Commit:** `a6a3700`
- **Gotchas:** los assets SÍ existen en `frontend/public/legacy-assets/{svg,images}/` — no había necesidad de copiarlos. El problema era puramente en la detección del entorno.
- **Sentinel:** no aplica.
- **GLORY-RS:** el patrón de detección de entorno (usar `__TAURI_INTERNALS__` en vez de flags de aplicación) es exportable, pero el caso es específico de Kamples.

## 224A-1 — script de limpieza automática del target dir de cargo
- **Que:** el target dir compartido `C:\tmp\glory-target` (configurado en `.cargo/config.toml`) crecía sin lí­mite hasta 11+ GB por incremental cache + deps de versiones viejas. Se añade `scripts/clean-cargo-target.ps1` con tres modos: safe (default — borra `examples/`, `doc/` + `cargo sweep --time 7` si está instalado, NO toca incremental), `-Hard` (también borra `incremental/`), `-Aggressive` (también borra `deps/`, `build/`, `.fingerprint/`). Detecta y mata `glory-backend.exe` antes de tocar paths que pueden estar lockeados. Se añaden 3 npm scripts: `clean:cargo` (safe), `clean:cargo:hard` (aggressive), `dev:clean` (limpia + dev). Hook `postdev` corre la versión safe automáticamente al cerrar `npm run dev`. Se instala `cargo-sweep` globalmente para que la versión safe haga limpieza fina por edad (>7 dí­as).
- **Por que:** lección recurrente — el target dir crece silenciosamente hasta que ocupa decenas de GB. Sin un mecanismo automático, queda librado a "acordarse de limpiar". El hook `postdev` lo hace transparente sin penalizar el siguiente arranque (no toca incremental por default). `cargo-sweep` permite borrar artifacts viejos sin perder caché útil. La versión `-Hard`/`-Aggressive` queda explícita para casos en que se necesita recuperar disco a fondo.
- **Archivos creados:** `scripts/clean-cargo-target.ps1`.
- **Archivos modificados:** `package.json` (+`clean:cargo`, `clean:cargo:hard`, `dev:clean`, `postdev`).
- **Validacion:** ejecutado `clean-cargo-target.ps1 -DryRun` → reporta tamaños sin tocar; ejecutado `-Hard` real → recuperó 3.5 GB del incremental cache (de 11448 MB → 7944 MB). `cargo install cargo-sweep` exit 0.
- **Sentinel:** no aplica — script de tooling.
- **GLORY-RS:** no aplica — específico del workspace local.

# Tareas completadas — 2026-04-21

## 214A-1 — Volumen persistente para uploads en deploy Rust
- **Qué:** El template Docker (`rust-stack.yaml`) no persistía el directorio `uploads/` ni copiaba `content/` al runtime. Cada redeploy borraba las imágenes subidas por CMS.
- **Archivos:**
  - `coolify-manager-rs/templates/rust-stack.yaml` — Agregado `COPY content/`, `mkdir uploads/content`, volumen `uploads_data:/app/uploads`
  - `coolify-manager-rs/config/settings.json` — sourcePaths del sitio studio actualizados a `["/app/uploads"]`
  - `src/handlers/image_proxy.rs` — Logging diagnóstico: cuando una imagen no se encuentra, ahora logea el path resuelto y el error exacto
- **Gotchas:** Las imágenes ya perdidas en producción no se pueden recuperar (no hay backup previo). El volumen previene pérdidas futuras.
- **Sentinel:** No aplica — cambio de infraestructura
- **GLORY-RS:** No aplica — específico de deployment

## 214A-2 — SEO dinámico para crawlers (reemplaza prerender estático)
- **Qué:** El middleware de prerender (114A-SEO3) servía HTML estático generado con Puppeteer. El contenido de CMS (servicios, proyectos) es editable, por lo que los archivos estáticos se volvían obsoletos. Refactorizado a inyección dinámica de meta tags SEO desde la BD.
- **Archivos:**
  - `src/middleware/prerender.rs` — Reescrito: ahora inyecta title, description, OG, canonical y Twitter cards en el index.html del SPA. Rutas estáticas usan meta fijos, rutas dinámicas (`/servicios/:slug`, `/proyectos/:slug`) consultan la BD.
  - `src/handlers/mod.rs` — `create_app()` ahora pasa `PrerenderState` (pool + static_dir + app_url) al middleware con `from_fn_with_state`
- **Gotchas:** La tabla `projects` tiene campos `meta_title` y `meta_description` específicos para SEO que se usan con `COALESCE` sobre title/description.
- **Sentinel:** No aplica
- **GLORY-RS:** El middleware es específico de nakomi (rutas y meta hardcodeados). Si se generaliza el routing SEO, podría moverse a glory-rs en el futuro.

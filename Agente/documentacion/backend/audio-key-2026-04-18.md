# Audio Key — 2026-04-18

## Estado actual

- `174A-32`: existe `src/audio/key.rs` con detección de tonalidad por cromagrama FFT y perfiles major/minor.
- El detector funciona sobre muestras en memoria y sobre archivos reales vía Symphonia.
- El resultado actual devuelve `music_key` en rango `0..11`, `scale`, `confidence`, `analyzed_seconds` y el cromagrama agregado normalizado.

## Algoritmo implementado

- Ventana máxima de análisis: `30s`.
- Remuestreo determinista a `22.05 kHz`.
- Análisis espectral:
  - `FFT_WINDOW_SIZE = 4096`
  - `FFT_HOP_SIZE = 2048`
- Construcción de cromagrama:
  - suma de magnitudes por pitch class
  - comparación multi-octava por clase cromática para evitar sesgos en fronteras `C/B`
  - ponderación por `1 / frecuencia` para reducir el peso de armónicos altos
  - normalización L2 por frame y al final del acumulado
- Selección de tonalidad:
  - perfiles mayor/menor tipo Krumhansl rotados en 12 tónicas
  - similitud coseno contra el croma global
  - `confidence` basada en mejor score contra la media de candidatos

## API pública

- `detect_key(samples, sample_rate_hz) -> Option<KeyAnalysis>`
- `detect_key_from_file(path, format_hint) -> Result<Option<KeyAnalysis>, KeyError>`

## Notas de migración

- El plan pedía cromagramas + scale sin crates nuevos. La implementación usa solo `symphonia` y `rustfft`, por lo que mantiene el stack ya aprobado.
- La calibración de `confidence` sigue siendo pragmática, no musicológica. Si `174A-34` necesita umbrales operativos más finos, conviene guardar temporalmente `confidence` en metadata del pipeline para ajustar con datos reales.
- Igual que BPM, esta pieza es suficientemente agnóstica para considerar moverla a `glory-rs`, pero conviene hacerlo cuando `audio_pipeline.rs` estabilice el contrato compartido.

# Despliegue Kamples — gestión de secrets y persistencia

> **Tareas:** 174A-112 (Dockerfile backend) · 174A-113 (Dockerfile frontend) · 174A-114 (docker-compose) · 174A-115 (secrets)
> **Aplica a:** Coolify v4 + docker-compose.

## Reglas absolutas (anti-pérdida-de-datos)

> Esta sección existe porque ya se borró la base de datos de producción una vez
> en el pasado (ver nota Fase 19 del roadmap). Nada de lo siguiente es
> opcional.

1. **NUNCA** ejecutar `docker compose down -v` en producción. La flag `-v`
   borra los volúmenes nombrados del stack — incluido `kamples_postgres_data`.
   Para parar el stack: `docker compose down` (sin flags) o `docker compose stop`.
2. **NUNCA** renombrar un volumen ya creado. Si cambias el nombre en el YAML,
   docker crea un volumen nuevo VACÍO y deja el anterior huérfano. El siguiente
   `docker volume prune` lo borra. Los nombres actuales son `kamples_postgres_data`
   y `kamples_redis_data`.
3. **Antes de cualquier deploy o migración SQL grande**, ejecutar manualmente:
   ```bash
   ./scripts/backup-postgres.sh pre-deploy
   ```
   Eso genera `/data/kamples/db_dumps/kamples-<TS>-pre-deploy.dump`. Si algo
   sale mal, restaurar con:
   ```bash
   docker compose exec -T postgres pg_restore --clean --if-exists \
       -U $POSTGRES_USER -d $POSTGRES_DB < /data/kamples/db_dumps/<dump>.dump
   ```
4. **Bind-mount externo** `/data/kamples/uploads` → `/data/uploads` dentro
   del contenedor. Los uploads de samples viven en el filesystem del host.
   Aunque borres el contenedor o el volumen postgres, los archivos físicos
   sobreviven en `/data/kamples/uploads`. **NO mover esa ruta** sin migrar
   los archivos primero (`rsync -aAXv /data/kamples/uploads/ /nueva/ruta/`).
5. **Sidecar `postgres-backup`** corre `pg_dump` cada 6h y rota >14 días.
   Los dumps se escriben a `/data/kamples/db_dumps`. Configurar un job
   externo (rclone/restic) que copie ese directorio a un bucket
   off-site (S3/B2/Backblaze) **al menos una vez al día**. El sidecar no
   sustituye a un backup remoto.
6. La política `restart: unless-stopped` en `docker-compose.yml` hace que
   los contenedores se reinicien si crashean, pero **NO** si los detuviste
   manualmente con `docker compose stop`. Esto evita que un crash silente
   degenere en pérdida de datos por reinicios infinitos.

## Estructura de directorios en el host

```
/data/kamples/
├── uploads/        # bind-mount → /data/uploads (storage local del backend)
├── db_dumps/       # bind-mount → /dumps (sidecar de backup + manual)
└── app_logs/       # bind-mount → /app/logs (logs del backend si LOG_FORMAT=json no basta)
```

Permisos: el contenedor `app` corre como uid 1000 (usuario `app`). Los
directorios del host deben ser propiedad de uid 1000:
```bash
mkdir -p /data/kamples/{uploads,db_dumps,app_logs}
chown -R 1000:1000 /data/kamples
chmod 750 /data/kamples
```

## Secrets — gestión

### Dónde viven
- **En desarrollo local:** `glory-rust-template/.env` (gitignored).
- **En producción Coolify:** UI → Service → Environment Variables. Coolify
  los inyecta en `docker-compose.yml` vía `${VAR}`. NO commitear `.env`
  de producción.
- **El archivo `.env.production.example` del repo** documenta el formato y
  qué variables son requeridas vs opcionales. Es la referencia.

### Cómo generar valores fuertes

| Variable                       | Comando para generar                                      |
| ------------------------------ | --------------------------------------------------------- |
| `JWT_SECRET`                   | `openssl rand -base64 48`                                 |
| `WS_SECRET`                    | `openssl rand -base64 48` (o vacío → reusa JWT_SECRET)    |
| `POSTGRES_PASSWORD`            | `openssl rand -base64 32`                                 |
| `SCRAPER_SECRET`               | `openssl rand -hex 32`                                    |
| `VAPID_*` (web push)           | `npx web-push generate-vapid-keys`                        |
| `FCM_SERVICE_ACCOUNT_JSON`     | Descargar JSON de Firebase Console → pegar en una línea   |

### Rotación

- **JWT_SECRET / WS_SECRET:** rotar invalida todas las sesiones activas y
  todos los tickets WebSocket en vuelo. Los usuarios deberán re-loguearse.
  Ventana: anunciar y hacer en madrugada.
- **POSTGRES_PASSWORD:** requiere actualizar primero el rol en la BD
  (`ALTER USER ... WITH PASSWORD ...`) y LUEGO reiniciar el stack con la
  nueva variable. Si lo cambias en `.env` antes, el backend perderá la
  conexión.
- **STRIPE_*:** rotar desde el dashboard de Stripe; webhook secret aparte
  para el endpoint `/api/stripe/webhook`.
- **GOOGLE_CLIENT_IDS:** lista CSV. Añadir nuevos sin quitar viejos durante
  la transición; quitar viejos solo después de actualizar todos los clientes
  móviles.

### Variables que pueden faltar
Las variables marcadas opcionales en `.env.production.example` deshabilitan
silenciosamente el canal correspondiente cuando faltan (Web Push, FCM, SMTP,
Stripe). El backend loggea un warning al arranque pero sigue funcionando.

## Build de imágenes

### Backend (Rust)
```bash
docker build -t glory-backend:latest .
```
- Build usa `SQLX_OFFLINE=true` y lee `.sqlx/`. Si modificaste un
  `sqlx::query!` debes correr `cargo sqlx prepare` y commitear `.sqlx/`
  ANTES del build. Si no, el build falla con "no cached data for this query".
- Tiempo: ~5-10 min en cold cache, ~1-2 min con cache de cargo registry.

### Frontend (Vite + nginx)
```bash
# 1) Generar el cliente Orval contra el backend (requiere openapi.json
#    actualizado en la raíz del repo).
cargo run -- --emit-openapi openapi.json

# 2) Build
docker build -t glory-frontend:latest -f frontend/Dockerfile frontend/
```

## Deploy con Coolify Manager

> Reglas: el deploy SIEMPRE pasa por `coolify-manager-rs`. Nunca SSH directo
> + docker-compose. (Ver protocolo regla 19 + 174A-117 — bloqueada hasta
> aprobación explícita del usuario).

```bash
# Secuencia obligatoria:
./scripts/backup-postgres.sh pre-deploy        # 1. backup manual
coolify-manager redeploy --name kamples        # 2. trigger build + redeploy
coolify-manager health   --name kamples        # 3. verificar HTTP 200
coolify-manager logs     --name kamples        # 4. revisar logs
```

Si `health` falla:
```bash
coolify-manager logs --name kamples --tail 200      # diagnosticar
coolify-manager restart --name kamples              # reintento simple
# Si persiste, restaurar último dump:
coolify-manager exec --name kamples -- /bin/sh -c \
    'pg_restore --clean --if-exists -U $POSTGRES_USER -d $POSTGRES_DB < /dumps/<ultimo>.dump'
```
